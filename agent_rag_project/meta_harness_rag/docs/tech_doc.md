## 0. 单函数迭代实现时的顺序（重要）

你的场景是：**每次只给实现者一条「写一个函数」的指令**，实现者**看不到整份规格里其它行的全文**，**只能看到仓库里已经写好的代码**。在这种工作流下：

1. **跨章节、§7 主循环、以及 §5/§6 与 §3（`McpClient`/`Executor`）的回调关系**，依赖容易对不上，必须在每条任务里写清前置条件（见 **§0.1**、§1 见 **§0.2**）。
2. **推荐做法**：每一条「写函数」的指令里，由你（或 Harness）**显式附上**：`[步骤编号]`、`所属类/文件`、**必须先已存在的函数/类名**（即本函数在实现中会**直接调用或类型注解引用**到的符号，其实现须**早于本步**已在仓库中可用；若尚无则写「无，仅标准库」）、以及**本步目标函数的规格摘录**：**怎样声明**（`def` / `async def`、参数名与类型、是否可为 `None`）、**各参数含义**、**返回什么类型或结构**（可直接从下文 §1–§7 对应表格的「接口」「参数说明」「输出」三列复制，避免实现者猜）。不要假设实现者记得 §4 的 Planner 约束或 §7 的循环结构。
3. **`RagOrchestrator`（§7）和 `Generator`（§6）/ `Evaluator`（§5）怎么配合**  
   - **运行时**：`answer` 从子任务队列里取出一条 `task` 后，会调用 **`Generator.run_subtask(...)`**（§6）。在这一次子任务内部，`run_subtask` 会反复用 **§3.2 `Executor`** 调 MCP，并在适当时机调用 **§5 `Evaluator`**，细节见 §5、§6 表格。  
   - **「桩」是啥、为啥要提**：若你还**没**把 §5、§6 写完整，又想先打通 **`answer` 的大循环**（规划 → 出队 → 调子任务 → 再 `replan`），可以让 `Generator` / `Evaluator` 先做成**只占位、行为极简**的实现（例如用 **`typing.Protocol` 约定接口**，`run_subtask` 里**直接 `return` 一个写死的 `dict`**，键满足 §6 里 **`SubtaskResult` 约定**即可）。这样 `answer` 不会因为「子任务里缺方法」而跑不起来。  
   - **「§5 → §6」**：指写**真逻辑**时的推荐顺序——因为 **`run_subtask` 里会调用 `Evaluator` 的方法**，一般**先**按 §5 把评估写好，**再**按 §6 把生成器里循环写全，和 **§0.1** 表里的先后顺序一致。

## 1. MemoryManager 模块

> **§1.0 工具结果中「证据文本」统一抽取（兼容 `query_knowledge_hub` / `search_by_metadata` 与其它仅返回正文的工具）**  
> 写入短期记忆的 **`result: dict`** 常见两类：（**A**）含 **`result["citations"]`**（多来自 MCP **`structuredContent.citations`** 归并），条目中常有 **`text_snippet`**；（**B**）**无 `citations` 或列表为空**，正文在 **`result["text"]`**（或由 §3 `Executor` 事先归一化的等价顶层字段）。**单条对象 `c: dict`（常为 citation 一项）**：若 **`(c.get("text_snippet") or "").strip()`** 非空，则证据文本取 **`text_snippet`**；**否则**若 **`(c.get("text") or "").strip()`** 非空，则取 **`text`**；二者皆空则该条可不参与 embedding。**整条 `result`**：若 **`citations` 非空**，对其中每一项按上一句取串，得到多条证据；若 **`citations` 缺失或为空**，则证据文本为 **`(result.get("text") or "").strip()`**（视为**一条**合并正文，此时 `chunk_id`/`source` 可用空串或由 `Executor` 填工具名占位，以便仍能 `encode`）。**与下表的关系**：上述语义**落实为 §1 表中正式接口** **`_evidence_body`**、**`_evidence_chunks_from_result`** 两行（须在 **`compute_memory_similarity` / `add_short_term`** 等之前实现）；其它 §1 方法通过调用二者避免重复判断。

| 接口 | 目标/功能 | 技术选型/实现方案 / 开发步骤 | 参数说明 | 输出 | 伪代码 |
|-------|------------|------------------------------|-----------|-------------------|------|
| `__init__(self, long_term_collection=None, sqlite_conn=None, qdrant_collection=None, config: dict = None)` | 初始化 MemoryManager 类 | **开发步骤**：<br>1. 初始化短期记忆列表 `self.short_term = []`<br>2. 接收外部传入的长期记忆数据库ChromaDB实例 `long_term_collection`，存为 `self.long_term_collection`、SQLite实例 `sqlite_conn`，存为 `self.sqlite_conn`、Qdrant实例 `qdrant_collection`，存为 `self.qdrant_collection`<br>3. 配置默认参数或外部传入的 config（如短期记忆容量限制（可默认50条）、阈值 threshold、压缩截断N、权重 w_time/w_importance/w_freq、短期记忆长期记忆情景记忆在选取topk时候的各自的k值（可预设短期=5、长期=3、情景=2）<br>4. 初始化可能需要的 LLM 工具（使用 `src.libs.llm.llm_factory.LLMFactory.create` 并传入 `settings.yaml` 配置创建 LLM 实例）或 DenseEncoder 实例（通过mcp_rag.src.ingestion.embedding.dense_encoder.DenseEncoder创建）（compress_memory 内使用）<br>5. 所有其他函数使用到的属性在 init 中按需初始化或延迟初始化（如 timestamp、access_count 等） | long_term_collection: 外部 ChromaDB collection 实例；config: 可选配置字典 | None |

| `_evidence_chunks_from_result(self, result: dict) -> List[dict]` | 将整次工具 **`result`** 归一为 **`[{"chunk_id", "source", "text_snippet"}, ...]`**，供 **`add_short_term`**、**`compute_memory_similarity`**、**`promote_to_long_term`**（metadata **`chunks`** 推导）、**`add_event`**（**`related_chunks`**）等复用 | **开发步骤**：<br>1. **`citations = result.get("citations") or []`**。<br>2. 若 **`citations` 非空**：**`out = []`**；对其中每个 **`c`**： **`body = self._evidence_body(c)`**；若 **`body` 为空则 `continue`**；否则 **`out.append({"chunk_id": str(c.get("chunk_id") or ""), "source": str(c.get("source") or c.get("doc_id") or ""), "text_snippet": body})`**（**`doc_id`** 为可选别名，实现自定）。<br>3. 若 **`citations` 为空或不存在**：**`body = (result.get("text") or "").strip()`**；若 **`body` 为空则 `return []`**；否则 **`return [{"chunk_id": "", "source": str(result.get("tool") or result.get("tool_name") or ""), "text_snippet": body}]`**（**`tool` / `tool_name`** 由 §3 **`Executor`** 归一化时可选填入）。<br>4. 第 2 步结束时 **`return out`**。 | `result`：单次任务写入记忆的 `dict`（含 **`citations`** 或仅 **`text`**） | **`List[dict]`**，每项必含 **`chunk_id`**、**`source`**、**`text_snippet`** |

> **「LLM 跨章约定」**（§3.2–§7，与 §1 / §2 一致）  
> 凡规格中**需调用 LLM** 的类（**`Executor`**、**`PlannerAgent`**、**`Evaluator`**、**`Generator`**）：**禁止**在构造函数中从外部传入 **`llm` 实例**；须在各自 **`__init__(self, config: dict, ...)`** 内使用 **`src.libs.llm.llm_factory.LLMFactory.create`**，并传入 **`config` 中与 `agent_rag/config/settings.yaml` 对齐的配置**（通常使用合并后 `dict` 的 **`llm`** 键，形状与 §1/§2 所用相同）；若本类 **`__init__` 中已创建** `self._llm`，后续方法**直接复用**，**不重复 `create`**。调用 LLM 时一律用 **`src.libs.llm.base_llm.Message`** 构造消息列表（**`system_prompt`**、**`user_prompt`** 等），再调用 **`self._llm.chat(...)`**。  
> **回退**：**任何**「LLM `chat` 失败或超时」**不得**在规格层写死为**唯一**某种算法（例如**不得**规定「凡是自然语言失败一律 TextRank」）。须**按该接口的职责与输出形状**在**对应章节表格 / 实现**中**单独写明**回退策略（可为 **TextRank**、**规则截断**、**首段拼接**、**空串/占位符**、**重试后仍失败则抛错**等；§1 **`compress_memory`**、§2 **`get_relevant_context`** 中的 TextRank 仅为**该两接口的示例**，其它接口**照各自需求设计**）。输出**严格 JSON** 的路径（**`fill_arguments`**、**`plan`/`replan`**、**`evaluate`**、**`choose_next_action`** 等）——**`chat` 失败或超时**时采用 **规则兜底**、**固定空结构**或 **报错**（在**各方法**规格中写死一种或组合，整仓对「JSON 类」可统一风格，但**仍非**「一律 TextRank」）。

## 3. MCP 执行子系统

> **组成**：**§3.1 `McpClient`** 为 MCP 会话与所选客户端库之间的**薄适配层**，对外固定 **`call_tool` 的入参与返回形态**，便于 §3.2 `Executor` 及单测与具体 SDK 解耦；**`list_tools` 不在本层实现**，由 §7 `RagOrchestrator` 与会话侧约定统一调用并缓存。  
> **多模态（与 §1 无关）**：MCP **`CallToolResult.content`** 为**内容块列表**，可**同时**含 **`type: "text"`** 与 **`type: "image"`**（**`data`** 为 base64，**`mimeType`** 如 `image/png`）；检索类工具（如 **`query_knowledge_hub`**）在命中带图 chunk 时常见「正文 + 若干图块」。§3.1 **须原样保留**两类块于归一 **`content`**；§3.2 **透传**该 **`dict`**；§6 **中间链路仍只用文本摘要**，图块写入旁路 **`_inner_images`**；§7 **最终对用户**按需附带 **`images`**。**§1 `MemoryManager` 不存 base64，本章不修改 §1。**

### 3.0 官方 `mcp` Python SDK：客户端建连（实现参考）

本仓实现的是 **MCP 客户端**（调用已存在的 MCP server 上的工具）。stdio 场景下典型顺序：**`mcp.client.stdio.stdio_client`** + **`mcp.StdioServerParameters`**（描述子进程命令/工作目录等）得到 `(read, write)` → **`async with ClientSession(read, write) as session:`**（`mcp.client.session.ClientSession`，不少示例写作 `from mcp import ClientSession`）→ **`await session.initialize()`**。此后 **`session` 即 §3.1 的 `underlying_session`**：**`await session.list_tools()`** 仅由 **§7** 拉取并缓存；**`await session.call_tool(...)`** 由 **§3.1 `McpClient.call_tool`** 委托并做返回归一化。若不用 stdio，改用官方包提供的其它 transport，仍须在 **`initialize` 之后**再把会话交给 `McpClient`。传输与会话生命周期**不属于 §3.1**：由 **§7 `RagOrchestrator`** 在首个需 MCP 的 **async** 路径内（见 **§7.0**）完成；**不在同步 `__init__` 内**完成。

| `__init__(self, mcp_client: McpClient, config: dict = None)` | 初始化 `Executor`：在 **§3.1 `McpClient`** 之上封装 **`tools/call`** 及单工具参数填充；**不**决定全局工具调用顺序；不负责 `list_tools` 与工具列表缓存 | **开发步骤**：<br>1. `self._mcp = mcp_client`，**类型须为 `McpClient`**（§3.1），且其底层会话已完成 MCP `initialize`。<br>2. `self._config = config or {}`（超时、最大重试等）。<br>3. **`self._llm`**：在 **`__init__` 内**用 **`src.libs.llm.llm_factory.LLMFactory.create`** 并传入 **`config` 中 `llm` 段**（与 §1/§2 相同来源），**供 `fill_arguments` 专用**；**禁止**从外部再传 `llm`。<br>4. 不在本类内维护 `tools/list` 缓存；`inputSchema` 由 §7 经 `get_input_schema` 注入 | **`mcp_client`**：§3.1 **`McpClient`** 实例，其 **`call_tool`** 可被 **`await`**。**`config`**：可选 `dict`；**须**含 **`llm`**（`dict`，与 `settings.yaml` 顶层 **`llm:`** 同形）供 **`LLMFactory.create`**；**建议**含 **`rag_agent.executor`** 或等价扁平键（**MCP `tools/call` 重试**与 **`fill_arguments` 的 LLM 重试须分键**，勿混用同一语义）：**`max_retries`**（`int`，**再试**次数，**不含**首次 `call_tool`；默认如 `3`）、**`backoff_base_sec`**（`float`，**`call_tool`** 指数退避基数秒）；**`fill_arguments_max_llm_retries`**（`int`，**再试**次数，**不含**首轮 `chat`；默认如 `2`，即首轮 + 2 次重试共 3 轮）、**`fill_arguments_retry_backoff_sec`**（`float`，相邻两轮 `chat` 之间 sleep 秒数，默认如 `0.3`）；其它键实现自定。 | **`None`**。 |

| `async def call_tool(self, name: str, arguments: dict) -> dict` | 经 **`McpClient.call_tool`** 发起 **`tools/call`**，并对失败路径做重试策略 | **开发步骤**：<br>1. 校验 `name` 非空。<br>2. `await self._mcp.call_tool(name, arguments)`。<br>3. 若返回已含归一字段则直接使用；否则按 §3.1 约定补全后再判错。**多模态**：成功路径下 **`content` 内 text/image 块**与 §3.1 一致，**本类不拆图、不转码**，原样交给 §6。<br>4. 失败时依 `self._config` 做**指数退避**重试；超出上限则抛出异常或返回带 `isError` 的结构（实现须整仓统一） | **`name`**、**`arguments`**：同 §3.1 **`call_tool`**。 | **`dict`**：与 §3.1 归一形状一致（**可含 image 块**）；错误路径通常仅 **`TextContent`** 说明。 |

| `async def fill_arguments(self, tool_name: str, input_schema: dict, user_intent: str, prior_observation) -> dict` | 仅依据**当前工具**的 `input_schema` 生成 `arguments` | **开发步骤**：<br>1. **`Message`**：`system_prompt` 要求**只输出合法 JSON 对象**（无 markdown）；`user_prompt` 含 **`input_schema`**（仅此工具）、**`user_intent`**、**`prior_observation`**（无则为空字符串）。<br>2. **`self._llm.chat(...)`**（或异步 `chat`，与所选 `BaseLLM` 一致）→ `json.loads`；**`required` 校验**（`jsonschema` 或手写）。<br>3. **重试与失败判定（推荐默认，整仓写死）**：下列**任一**成立则计为**本轮失败**并进入**下一轮 `chat`**（直至用尽 **`fill_arguments_max_llm_retries`** 所指**再试**次数，**不含**首轮）：**`self._llm.chat` 抛错或超时**；返回体**无法解析为单个 JSON object**（含 markdown 代码围栏、截断、多段 JSON）；**`json.loads` 成功但对象不满足 `input_schema`**（缺 **`required`**、类型不符、若实现选择「拒绝额外键」时的非法键）。每轮重试前可按 **`fill_arguments_retry_backoff_sec`** 做短 **sleep**。**不得**与 **`max_retries` / `backoff_base_sec`**（**§3.2 `call_tool`** 的 MCP 层）混用同一套键或语义。**不须**与 TextRank 等摘要算法挂钩（见 **「LLM 跨章约定」**，§2 末 blockquote）。<br>4. **用尽重试仍失败**：本方法**抛异常**（如 **`ValueError`** 或自定义 **`ArgumentFillError`**），信息须含 **`tool_name`** 与末次校验/解析错误摘要；**不要**返回「半合法」`dict` 冒充成功。 | **`tool_name`**：`str`，与 **`tools/list`** 中 **`name`** 一致，用于日志/重试提示。**`input_schema`**：`dict`，通常为 JSON Schema 子集，至少含 **`type":"object"`**、**`properties`**、**`required`**（`list[str]`，可无表示无必填）。**`user_intent`**：`str`，本子任务自然语言描述（多来自 **`task["description"]`**）。**`prior_observation`**：`str` 或 **`None`**；为上游压缩后的 MCP/评估短文本，**`None`** 时按空串参与 prompt。 | **`dict`**（**仅成功路径**）：须**满足** **`input_schema`** 中 **`required`** 所列键；值类型与 **`properties`** 中声明一致；**不得**含 schema 未声明且实现选择「拒绝额外键」时的多余键（整仓统一）。**失败路径**：见上**步骤 4**，以**异常**退出，由 **`execute_task`** 捕获并转为归一错误 **`dict`**；或 **`raise`** 由 **`Generator`** 记入 **`self._inner_trace`**（与下行及 §6 整仓统一）。 |

| `async def execute_task(self, task: dict, get_input_schema) -> dict` | 执行单条子任务对应的「一步」MCP：`tool_name` → schema → 填参 → `call_tool` | **开发步骤**：<br>1. 从 `task` 取 `suggested_tool`（或上层已写入的 `tool_name`）。<br>2. `schema = get_input_schema(tool_name)`（§7 编排层提供：来自 `tools/list` 缓存的 `inputSchema`）。<br>3. **`try`**：`args = await self.fill_arguments(tool_name, schema, user_intent=task["description"], prior_observation=(task.get("prior_observation") or "")`（**默认**采用参数说明中 **(a)**：由 §6 写入 **`task["prior_observation"]`**；**若整仓选 (b)** 则改为使用 **`execute_task`** 形参 **`prior_observation`**，见参数说明）；**`except`** 捕获 **`fill_arguments`** 在**用尽 LLM 重试**后抛出之异常（见上行）：**不得**静默吞掉；**`return`** 与 §3.1 归一形状一致且 **`isError=True`** 之 **`dict`**（**`content`** 含可读错误，说明填参失败），**不再**调用 **`call_tool`**；或**整仓统一**为向 §6 暴露可记录之失败（如再 **`raise`** 由 **`Generator`** 记入 **`self._inner_trace`** 一条 **`ok=False`** 步）——**二选一并文档化**，与 §6 **`run_subtask`** 约定一致。<br>4. **`return await self.call_tool(tool_name, args)`**。 | **`task`**：`dict`，**至少**含 **`description`**（`str`）；**建议**含 **`suggested_tool`** 或 **`tool_name`**（`str`）其一；可有 **`id`**、**`intent`** 等 §4 产出字段。**`prior_observation` 传入本行的方式（整仓二选一写死）**：**(a)** §6 **`run_subtask`** 在每轮调用 **`execute_task`** 前写入 **`task["prior_observation"]`**（`str`，**首轮**为 **`""`**，后续轮为 **`_format_trace(self._inner_trace)`** 或等价短摘要）；本步写 **`prior_observation=task.get("prior_observation")`**，**`None`** 时按 **`""`** 传入 **`fill_arguments`**。**(b)** 将 **`execute_task`** 签名为 **`execute_task(self, task, get_input_schema, prior_observation: str | None = None)`**，由 **`run_subtask`** 显式传入；§6 表须同步。**`get_input_schema`**：可调用对象 **`(tool_name: str) -> dict`**（**同步**；若实现为 async 则整仓改为 **`await get_input_schema(...)`** 并在此表注明）；入参为 MCP 工具名，返回该工具的 **`inputSchema`** 对象（`dict`），不存在时抛 **`KeyError`/`ValueError`**（与 §7 约定一致）。 | **`dict`**：**正常路径**与 §3.1 **`call_tool`** 成功返回相同；**`fill_arguments`** 失败且本方法选择 **`return`** 错误结构时，为归一 **`isError=True`**（**未**发起 MCP **`tools/call`**）；**`call_tool`** 返回错误结构时原样向上返回。 |

**§3.2 内建议实现顺序**：`__init__` → `call_tool` → `fill_arguments` → **`execute_task` 最后**（在 §3.1 已就绪的前提下）。

**类型注解**：`McpClient` 与 `Executor` 若分属不同模块，对 `McpClient` 的前向引用可采用字符串形式 `"McpClient"` 或 `from __future__ import annotations`（实现自定）。

| `replan(self, query: str, context: str, tool_index: str, routing_hint: str, observation: str) -> list[dict]` | **Solve 后回灌**：根据**全局/子任务**执行观测修订或追加子任务；通过**改写检索问句、换工具、补步骤**改善证据 | **开发步骤**：<br>1. **`Message`**：<br>&nbsp;&nbsp;— **`system_prompt`**：在 **`plan`** 同形 JSON 数组输出基础上，**额外**写明 **【replan 修订策略】**（须整仓写死，可放 **`config["planner"]["replan_system_prompt"]`**）：<br>&nbsp;&nbsp;&nbsp;&nbsp;**（a）读 observation**：区分**子任务级**失败（某条 `needs_replan`）与 **§7 全局终稿门禁**（`_check_global_answer_readiness` 判定证据不足以回答**原问题**）。<br>&nbsp;&nbsp;&nbsp;&nbsp;**（b）换检索问句/关键词**：在新子任务 **`description`** 中给出**更可检索**的英文术语 + 中文关键词；可改 **`suggested_tool`** 或 **`filters`** 语义（写在 description，由 §6 填参落实）。**`query_knowledge_hub` / `search_by_metadata`** 调用 MCP 后，问句扩写（MQE/HyDE）由 **MCP 服务端** `query_expansion` **自动完成**，**replan 不得**输出或建议 MQE/HyDE、HyDE 段落、多查询扩展等（**不在 Harness/Planner 职责内**）。<br>&nbsp;&nbsp;&nbsp;&nbsp;**（c）换工具或补步骤**：如补 `get_document_full_text`、`search_by_metadata`（收窄 `doc_id`）、`get_neighbor_chunks` 等；**勿**重复已失败且无新信息的同参调用。<br>&nbsp;&nbsp;&nbsp;&nbsp;**（d）可选任务字段**：除 **`plan`** 必填键外，允许 **`retrieval_hints`**（`list[str]`，短句，如「改写 query 为…」「补某 doc 全文」）——**不得**含 MCP **`arguments`**，**不得**提及 MQE/HyDE。<br>&nbsp;&nbsp;— **`user_prompt`**：**`plan`** 之四节 + 第五节 **「执行与评估观测」**（粘贴 **`observation`**，控长如 ≤8k 字符）；仍**禁止** `inputSchema`。<br>2. **`self._llm.chat`** → 同 **`plan`** 解析与校验。<br>3. 返回新队列；§7 将新项 **`extend`** 或 **`extendleft`**（写死一种）后继续 **`run_subtask`**，直至全局门禁通过或达 **`max_global_replan_rounds`**。 | **`observation`**：`str`；来自 §6 **`observation_for_replan`** 拼接、§5 **`to_planner_observation`**、和/或 §7 **`GlobalAnswerReadiness.observation_for_replan`**（**建议**合并去重）。**控长**由 §7 **`global_replan_observation_max_chars`** 或本方法入参前截断。 | **`list[dict]`**：同 **`plan`**；新队列**仅含待执行**的追加/替换子任务（勿重复已完成且无需重做的项，实现自定）。 |

## 5. Evaluator 类（子任务内评估 / 内循环）

> **职责**：在 **单条子任务** 边界内，根据 **全局用户问题** `global_query`、子任务描述、已执行 MCP 的**短摘要**（非全量原始 JSON）、以及 §6 `Generator` 的草稿，判断**证据是否足够**、**是否通过**、是否建议 **`needs_replan`**；输出 **`EvalResult`**，并可压成 **`observation_for_replan`** 交给 §4 `PlannerAgent.replan`。  
> **`EvalResult` 约定**（`evaluate` / `quick_rule_check` 的返回值，键名实现时写死）：**`passed`**（`bool`）、**`score`**（`0~1` `float`）、**`require_more_tools`**（`bool`）、**`status`**（`"ok"` \| `"needs_replan"` \| `"hard_fail"`）、**`issues`**（`str`，短说明）。**字段分工**：**`passed`** = 仅就**当前子任务**，证据+草稿是否达到 **`done_criteria` / `description`**（**不**表示全局问题已答完）；**`require_more_tools`** = 是否建议**在本子任务内**再调 MCP；**`status`** = 局面分类（正常 / 要重规划 / 硬失败），**与 `passed` 独立**。§6 **`run_subtask`** 消费：**`passed==True` 且 `require_more_tools==False`** → 可提前 **`stop`**；**`status=="needs_replan"`** → **`SubtaskResult.status="needs_replan"`**。**具体判定标准须在 `evaluate` 的 `system_prompt` 中逐条写死**（见下行 **`evaluate`** 表），实现时写入 **`config["evaluator"]["eval_system_prompt"]`**（或等价键），**不得**只输出 JSON 键名而不写选择规则。

| 接口 | 目标/功能 | 技术选型/实现方案 / 开发步骤 | 参数说明 | 输出 |
|------|------------|------------------------------|-----------|------|
| `__init__(self, config: dict = None)` | 初始化 Evaluator | **开发步骤**：<br>1. `self._config = config or {}`（**`eval_max_chars`**、**`strict_json`**、**`eval_system_prompt`**（**`str`**，正文取自本表 **`evaluate`** 之 **【输出格式】~【一致性自检】**，可再拼 **`user_prompt_template`**）。<br>2. **`self._llm`**：在 **`__init__` 内**用 **`src.libs.llm.llm_factory.LLMFactory.create`** 并传入 **`config` 中 `llm` 段**；**禁止**从外部传入 `llm`。<br>3. 不调用 MCP、不持有 §3.2 `Executor` 或 §3.1 `McpClient` | **`config`**：见左；**`eval_system_prompt` 为空时**实现**不得**仅用「输出 JSON」敷衍，应内嵌与 **`evaluate`** 表等价的默认条文。 | **`None`**。 |

| `quick_rule_check(self, task: dict, tool_trace_summary: str)` | **可选**纯规则预检：在不调 LLM 时快速返回 `EvalResult` 或 `None`（表示交给 LLM） | **开发步骤**：<br>1. **示例硬规则**（实现可扩展，**须**返回完整 **`EvalResult` 五键）：若 **`tool_trace_summary`** 中 **`[error]`** 连续 **`n`** 次（**`n`** 由 **`config`**（如 **`rag_agent.evaluator.quick_rule_error_streak`**），默认 **3**）→ **`status="hard_fail"`**，**`passed=False`**，**`require_more_tools=False`**，**`score=0.0`**，**`issues`** 说明连续 MCP/填参失败。<br>2. 未命中任何硬规则 → 返回 **`None`**。<br>3. **调用点（须与 §6 对齐）**：由 **§6 `run_subtask` 步骤 3** 在 **`await evaluator.evaluate(...)` 之前**同步调用；返回非 **`None`** 时**跳过** **`evaluate`** 内之 **`self._llm.chat`**。<br>4. **与 `evaluate` 内「可选规则兜底」分工**：本方法只做**可短路整次评估 LLM** 的硬条件；**勿**与 **`evaluate`** 步骤 3 后置补丁重复同一 **`if`**（整仓写清）。 | **`task`**：子任务 **`dict`**（规则扩展时可读 **`intent`**、**`replan_triggers`** 等）。**`tool_trace_summary`**：与 **`evaluate`** 第三参**须同源**——§6 为当轮 **`self._format_trace(self._inner_trace)`**（或等价短摘要），**非** MCP 全量 JSON。 | **`dict`**（完整 **`EvalResult`**）或 **`None`**；**`None`** 表示调用方应继续走 **`evaluate`**。 |

| `evaluate(self, global_query: str, task: dict, tool_trace_summary: str, draft_answer: str) -> dict` | 对当前子任务状态做**一次性或周期性**评估，返回 **`EvalResult`** | **开发步骤**：<br>1. **`Message`**：<br>&nbsp;&nbsp;— **`system_prompt`**（**实现时必须写全下列选择标准**；可整段放 **`config["evaluator"]["eval_system_prompt"]`**，代码仅做占位符替换。**除格式约束外，下列条文须原样或等价语义出现在 prompt 中**，供后续 LLM 实现直接拷贝）：<br>&nbsp;&nbsp;&nbsp;&nbsp;**【输出格式】** 只输出**一个**合法 JSON 对象，**无** markdown 围栏、**无**前后说明；键**仅** **`passed`**、**`score`**、**`require_more_tools`**、**`status`**、**`issues`**；**`score`** 为 **0~1** 数字（可小数）。<br>&nbsp;&nbsp;&nbsp;&nbsp;**【评估范围】** 为空时，**`description`** 所要求的信息在草稿中**已覆盖**且无与证据明显矛盾的断言；**(c)** 无「证据不足却做肯定结论」；**(d)** 若 **`intent`** 为检索/查证类且 **`tool_trace_summary`** 为「（无）」或无任何成功工具行，**不得** **`passed=True`**。否则 **`passed=False`**。<br>&nbsp;&nbsp;&nbsp;&nbsp;**【`require_more_tools`】** **`True`**：本子任务**仍缺关键证据**，且**再调 1~2 次 MCP**（换工具/换 query/补 metadata）**有望**补齐——例如：未检索、检索空但 criteria 要求「至少一条来源」、仅有大纲缺正文、草稿写「待查」、**`suggested_tool`** 尚未尝试且与缺口匹配。**`False`**：已足够作答本任务，或**再调工具也难改善**（重复失败、题意已饱和、纯归纳/格式任务无需新证据）。**与 `passed` 可组合**：常见 **`passed=False, require_more_tools=True`**（继续内循环）；**`passed=True, require_more_tools=False`**（可结束子任务）。**避免** **`passed=True` 且 `require_more_tools=True`**（除非整仓明确允许「过关但仍建议加证据」，默认**禁止**）。<br>&nbsp;&nbsp;&nbsp;&nbsp;**【`status` 优先级】** 先定 **`status`**，再与其它字段对齐：**`hard_fail`** &gt; **`needs_replan`** &gt; **`ok`**。<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;— **`hard_fail`**：连续/致命工具失败（摘要含 **`[error]`** 且无法通过改参修复）、子任务目标**自相矛盾**、所需工具**不在**可用列表且无法替代、**`draft_answer`** 与 **`tool_trace_summary`** 中**明确事实**严重冲突且无法在本任务内调和。**此时** **`passed=False`**，**`require_more_tools=False`**，**`score`** 建议 **≤0.2**。<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;— **`needs_replan`**：**命中** **`task.replan_triggers`** 任一条（与 §4 列表语义一致，如「多轮检索仍空」「需跨文档对比但当前切片只覆盖单篇」「表/图证据缺失且 criteria 要求图表」）；或 **`intent`/描述** 与已执行工具**明显不匹配**（应用 **`search_by_metadata`** 却全程 **`query_knowledge_hub`**）；或 **`done_criteria`** 在当前子任务边界内**不可完成**（缺依赖子任务、范围定错）。**此时** **`passed=False`**，**`require_more_tools`** 通常 **`False`**（改由 §4 重规划，而非内循环加工具）；**`score`** 建议 **0.2~0.5**。<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;— **`ok`**：未触发 **`hard_fail`** / **`needs_replan`**；局面可由**继续工具**或**结束子任务**处理。**`status=ok`** 时 **`passed`** / **`require_more_tools`** 按上两条独立判定。<br>&nbsp;&nbsp;&nbsp;&nbsp;**【`score`】** 子任务完成度 **0~1**（与 **`passed`** 一致方向，可更细）：**≥0.8** 且 **`passed=True`**；**0.5~0.8** 部分满足；**&lt;0.5** 基本未满足或仅泛泛而谈。**`hard_fail`** 宜 **≤0.2**；**`needs_replan`** 宜 **0.2~0.5**。<br>&nbsp;&nbsp;&nbsp;&nbsp;**【`issues`】** 一句或几句中文（与 **`user_prompt`** 语言一致则全仓统一）：**缺什么** / **哪条 criteria 未满足** / **为何 `needs_replan` 或 `hard_fail`**；**禁止**复述整段草稿；**≤200 字**（实现可在解析后截断）。<br>&nbsp;&nbsp;&nbsp;&nbsp;**【一致性自检】** 输出前检查：**`hard_fail` 或 `needs_replan` ⇒ `passed=False`**；**`passed=True` ⇒ `status` 不得为 `hard_fail`**；**`needs_replan` ⇒ `require_more_tools` 宜为 `False`**（除非文档明确例外）。<br>&nbsp;&nbsp;— **`user_prompt`**（须整仓写死骨架，可放 **`config["evaluator"]["user_prompt_template"]`** 等）：按**固定小节**拼接，**小节标题**中英择一但整仓统一，**节与节之间空一行**，**自上而下**顺序为：<br>&nbsp;&nbsp;&nbsp;&nbsp;（1）**全局用户问题**：粘贴 **`global_query`** 原文（**非空**时不得用占位替换；若实现允许空 query，空则写「（无）」）。<br>&nbsp;&nbsp;&nbsp;&nbsp;（2）**当前子任务（§4 产出）**：逐项可读文本粘贴（**不得**把整包 **`task`** 原始 JSON 无说明地扔进模型，除非整仓明确采用「JSON §4 **`plan`** 输出类型一致：若为 **`list`** 则按行或编号拼接为可读段落，无则「（无）」）；**`replan_triggers`**（`task.get("replan_triggers")`，同上，无则「（无）」）。**`task` 中其它键**（如 **`description`** 的中段或尾部并保留首句；**尽量不动** (1) **全文**；截断处可加 **`…[truncated]`** 标记并**建议**打日志）。<br>2. **`self._llm.chat`** → **`json.loads`**；缺键时补默认（**`passed=False`**、**`score=0.0`**、**`require_more_tools=True`**、**`status="ok"`**、**`issues=""`**——**默认策略须整仓统一**）；**若 `strict_json` 为真**，解析失败或缺**必填**键则视为评估失败路径（返回固定 **`EvalResult`** 或抛错，**二选一整仓统一**）；失败/超时按 **「LLM 跨章约定」**（§2 末 blockquote，JSON 类）。<br>3. **可选规则兜底**（**勿**与 **`quick_rule_check`** 重复）：例——**`tool_trace_summary`** 为空且 **`task.intent`** 含 retrieve → **`require_more_tools=True`**、**`passed=False`**；解析后若违反 **【一致性自检】**，代码**覆写**对应字段并打日志。 | **`global_query`**：原始用户问题，`str`。**`task`**：当前子任务 **`dict`**，**至少**含 **`description`**、**`intent`**；**可选**含 **`id`**、**`suggested_tool`**、**`done_criteria`**、**`replan_triggers`**（类型与 §4 **`plan`** 一致）、**`depends_on`** 等——**`user_prompt`** 第 (2) 节**须**显式展开 **`done_criteria`** / | `to_planner_observation(self, eval_result: dict, max_chars: int = None) -> str` | 将 `EvalResult` 压成 **§4 `PlannerAgent.replan` 可用的短 `observation`** | **开发步骤**：<br>1. 拼接 `status`、`issues`、`require_more_tools` 为一段英文或中文短句（与 Planner prompt 语言一致，实现写死）。<br>2. 按 `max_chars` 截断 | **`eval_result`**：与 **`evaluate`** 返回值同形之 **`dict`**。**`max_chars`**：可选 `int`；**`None`** 时用 **`self._config.get("eval_max_chars")`** 或默认上界。 | **`str`**：供 **`replan(..., observation=...)`**；**单行或多句短文本**，无 JSON 包装。 |

## 6. Generator 类（子任务内求解 / 内循环）

> **职责**：消费 **§4 Planner 的一条子任务** `task`（`dict`）；在子任务边界内多次调用 **§3.2 `Executor`**（经 `execute_task` 或等价路径）访问 MCP 工具；按需调用 **§5 `Evaluator`**；产出 **`SubtaskResult`**（见下），供 §7 汇总、`replan` 的 `observation`、以及最终写回 §1/§2 的素材。  
> **`SubtaskResult` 约定**（`run_subtask` 的返回值，键名实现时写死）：**`task_id`**（`str`，**必填**，与 §4 Planner 本条子任务 **`task["id"]`** **完全一致**，供 §7 **`subtask_results`** 排序与证据包分段；**无**则 §7 **`_build_final_evidence_bundle`** 仅能按 **append 顺序** 遍历）、`status`（`"success"` \| `"failed"` \| `"needs_replan"`）、`draft_text`（本子任务侧草稿/摘录，可为空；**来源**为 **`draft_partial_answer` 返回的 `str`**，见该行 **`system_prompt` 约定**）、`tool_trace`（`list[dict]`，每步含 `tool_name`、`ok`、`summary`；**可选** `image_count: int` 表示该步 MCP 返回中的图块数，**不含** base64）、`observation_for_replan`（短字符串，供 `PlannerAgent.replan`）、`citations`（可选，`list[dict]`，与 §1 `add_short_term` 的 `result["citations"]` 形状对齐则最省事）、**`images`**（**可选** `list[dict]`，本子任务从 MCP **`content`** 收集的图，见 **`_collect_images_from_raw`**；无图则 **`[]`** 或省略）。  
> **`NextActionResult` 约定**（**`choose_next_action`** 返回值）：**`action`**（`"call_tool"` \| `"stop"` \| `"replan"`）、**`tool_name`**（`call_tool` 时非空且须在 **`tool_names`** 列表内）、**`reason`**（非空 `str`）。**不**含 MCP **`arguments`**。  
> **`self._inner_trace`（实例字段，整 §6 须一致）**：**本子任务内**已执行 MCP 的**文本轨迹**（`list[dict]`，元素形状同 **`SubtaskResult["tool_trace"]`**）。**`self._inner_images`（实例字段）**：**本子任务内**从各步 **`raw["content"]`** 抽取的**图片旁路**（**不**写入 **`summary`** 字符串、**不**进入 §5 **`evaluate`** 的 **`tool_trace_summary`**）。**`Generator.__init__`** 置 **`self._inner_trace = []`**、**`self._inner_images = []`**；**`reset_subtask_state`** 二者均清空；**`run_subtask`** 每步 MCP 后：**`append`** trace 条 + **`extend`** **`_collect_images_from_raw(raw, tool_name)`** 至 **`_inner_images`**；**`choose_next_action` / `_format_trace` / `Evaluator` / `prior_observation`** **仅**读 **`_inner_trace`**；**`return`** 时 **`tool_trace=list(self._inner_trace)`**、**`images=list(self._inner_images)`**（均为浅拷贝）。**§1 记忆仍只消费文本/citations，不读 `images`。**

| `async def run_subtask(self, global_query: str, task: dict, executor: Executor, get_input_schema, evaluator: Evaluator, tool_names: list = None) -> dict` | **子任务主循环**：执行 MCP → 摘要 →（可选）草稿 → §5 评估 → 决策下一步，直到 `stop` / 失败 / `needs_replan` | **开发步骤**：<br>1. `reset_subtask_state()`（清空 **`self._inner_trace`** 等，见上 **`self._inner_trace` 约定**）。<br>2. 循环：**`await self.choose_next_action(global_query, task, self._inner_trace, last_eval=..., tool_names=...)`**（**`tool_names`** 由 §7 传入，为 **`list(self._tools_by_name.keys())`** 或子集；首轮 **`last_eval`** 可为 **`None`**；冷启动见 **`choose_next_action`** 步骤 5）。若为 `call_tool`：**（与 §3.2 `execute_task` / `fill_arguments` 衔接）** 在每次 **`await executor.execute_task(task, get_input_schema)`** 之前，按 §3.2 **默认 (a)** 赋值 **`task["prior_observation"] = "" if not self._inner_trace else self._format_trace(self._inner_trace)`**（仅用**当前已累积**的 **`self._inner_trace`**）。再 **`raw = await executor.execute_task(...)`**；**不论** **`raw`** 为 **`call_tool`** 成功、**`call_tool`** 返回 **`isError=True`**，还是 **`fill_arguments`** 失败后 **`execute_task`** 按 §3.2 **`return`** 的归一 **`isError=True`**（**未**发起 MCP **`tools/call`**），**均须**经 **`summarize_mcp_result(raw)`** 得 **`summary`**；**`imgs = self._collect_images_from_raw(raw, tool_name)`**，**`self._inner_images.extend(imgs)`**；**`append`** **`{"tool_name": tool_name, "ok": not raw.get("isError", False), "summary": summary, "image_count": len(imgs)}`**（**`image_count` 可选**）至 **`self._inner_trace`**。**若整仓选用 §3.2 `execute_task` 之 (b)**，则不得写 **`task["prior_observation"]`**，须同步 **`Executor.execute_task`** 签名。<br>3. 每步或每隔 k 步：先 **`tool_trace_text = self._format_trace(self._inner_trace)`**（与同轮 **`evaluate(..., tool_trace_summary=...)`** **须**为**同一字符串**）。**若 §5 实现了 `quick_rule_check`**：`rule_hit = evaluator.quick_rule_check(task, tool_trace_text)`（**同步**，非 **`async`**）。**若 `rule_hit is not None`**：令 **`eval_result = rule_hit`**，**不得**再 **`await evaluator.evaluate(...)`**；**`draft`** 整仓写死其一：**`""`**、**沿用内层循环上一轮** **`draft`**、或**仍** **`await self.draft_partial_answer(global_query, task, tool_trace_text)`**（**三选一**，须与提前 **`stop`/`needs_replan`** 逻辑一致）。**若 `rule_hit is None` 或未实现 `quick_rule_check`**：**`draft = await self.draft_partial_answer(global_query, task, tool_trace_text)`**；**`eval_result = await evaluator.evaluate(global_query, task, tool_trace_text, draft)`**。此后：若 **`eval_result["require_more_tools"]`** 为假且 **`eval_result["passed"]`** 为真可提前 **`stop`**（策略写死）。**`evaluate`** 内**可选**规则兜底（§5）仅作用于**已调用** **`evaluate`** 之分支；与 **`quick_rule_check`** **勿**维护两套完全相同的条件。<br>4. 若 **`action=="replan"`** 或 **`eval_result["status"]=="needs_replan"`**：组装 **`SubtaskResult`**，**`status="needs_replan"`**，**`observation_for_replan = evaluator.to_planner_observation(eval_result)`**。<br>5. 正常结束：**`status="success"`** 或证据不足时 **`"failed"`**，填入 **`draft_text`**、**`citations`**（从 MCP 结构化结果提取，能取则取）。<br>6. **`return`** 符合 **`SubtaskResult` 约定** 的 **`dict`**：**`task_id=str(task["id"])`**（**必填**；若 **`task`** 缺 **`id`** 则实现抛错或用队列序号兜底并打日志，**整仓写死一种**）、**`tool_trace=list(self._inner_trace)`**、**`images=list(self._inner_images)`**（均为浅拷贝），**不得**直接返回内部列表引用。 | **`global_query`**：`str`，与 §7 **`answer`** 的 **`query`** ## 7. RAG Agent 编排层 — `RagOrchestrator` 类

> **`AnswerResult` 约定**（**`answer`** 面向用户的返回值，与 §6 子任务内纯文本轨迹分离）：**`text`**（`str`，最终回答正文，可 Markdown）；**`images`**（**可选** `list[dict]`，每项 **`mime_type`** + **`data`**（base64），可选 **`caption`** / **`source`** / **`tool_name`**）。无图时 **`images=[]`** 或省略键。**§1 `add_short_term` 仍只写入文本侧 `result_dict`（citations 等），不把 `images` 存入 §1。**

> **`GlobalAnswerReadiness` 约定**（**`_check_global_answer_readiness`** 返回值，键名写死）：**`sufficient_for_answer`**（`bool`，就**全局** `query` 而言证据是否足以撰写最终答案）；**`need_replan`**（`bool`，**`True`** 时 §7 **不得**进入终稿合成，须 **`replan`**）；**`issues`**（`str`，缺什么、哪类子任务失败、与 **`query`** 哪些要点未覆盖）；**`observation_for_replan`**（`str`，供 **`PlannerAgent.replan`** 的第五节观测，**须**含可执行修订建议）；**`suggested_retrieval_changes`**（**可选** `list[str]`，短句列表，如「改写检索问句为…」「对 doc_X 补 `get_document_full_text`」「收紧 `filters.has_table`」；**不含** MQE/HyDE，见 §4 **`replan`**）。**与 §5 区别**：§5 评**单条子任务**；本结构评**全局**是否可作答。

> **说明**：本节描述的是 **RAG Agent 本体**的主循环与 MCP 资源管理（缓存 `tools/list`、为 §4 Planner 构造全量工具索引、为 §3.2 `Executor` 经 `get_input_schema` 提供 `inputSchema`）。**实现代码预期落在 `meta_harness_rag` 仓库**（Harness 进程、控制器、评测脚本可与编排同仓），类名可随仓库约定调整；本文以 `RagOrchestrator` 指代该职责边界。**对外入口**：**`RagOrchestrator(config, ...)`**（**`...`** 见 §7 表 **`__init__`** 可选存储句柄）——在 **`__init__` 内**根据 `config`（及可选句柄）**构造** §1 `MemoryManager`、§2 `ContextManager`、§4 `PlannerAgent`、§5 `Evaluator`、§6 `Generator`（各 § 表；其中 §4–§6 的 **LLM 均在各类自身 `__init__` 内 `src.libs.llm.llm_factory.LLMFactory.create`**，见 **§2 末「LLM 跨章约定」**）；**不**在同步 `__init__` 内建 MCP 传输或 **`ClientSession`**（缺 `await`）。**MCP 建连、`initialize`、`McpClient`、`Executor`** 在 **`answer`（或 **§7.0 (ii)** 之编排器 async 生命周期）**内完成；**`Executor(..., config=self._config)`** 与 §4–§6 **各自**在 **`__init__` 内**对 **`config["llm"]`** 调用 **`src.libs.llm.llm_factory.LLMFactory.create`**（通常得到**彼此独立**的 LLM 实例；若实现采用单例工厂可文档化说明）。Harness 侧通常 **`asyncio.run(orchestrator.answer(query, session_id=...))`**，得到 **`AnswerResult`**（**`text` + 可选 `images`**）；**`session_id`** 可选。**§4 Planner、§5 Evaluator 无需为多模态改表**（仍消费文本摘要）。

### 7.0 MCP 与会话：`__init__` 与 async 首连的分工

**同步 `__init__(config[, 可选存储句柄])` 内**：见 §7 表 **`RagOrchestrator.__init__`** **开发步骤**；在此 **new** §1 **`MemoryManager`**、§2 **`ContextManager`**、§4–§6（**不依赖** MCP 会话）；§4–§6 内部各自 **`LLMFactory.create`**，见 **「LLM 跨章约定」**。初始化 **`self._tools_by_name`**、**`self._mcp_session`**、**`self._executor`**。

**async 首连与生存期（须满足 SDK）**：**`ClientSession` 在 `async with` 结束后即不可用**，因此 **§3.0 建连 → `ClientSession` → `await initialize()` → `McpClient` → `Executor` → `list_tools` 写入 `_tools_by_name`** 必须发生在 **`answer` 全文**所进入的 **`async with stdio_client(...):` `async with ClientSession(...):`**（或编排器 **`async with orchestrator:`** / **`__aenter__`/`__aexit__`** 等**等价长生命周期**）**之内**，且该作用域**须覆盖**后续所有 **`run_subtask` → `Executor.call_tool`**。**禁止**在已退出的 `async with ClientSession` 体之后再调用 **`self._mcp_session.call_tool`**。**`_ensure_tools_cache`**：仅在该**已打开**的 MCP 上下文中调用（或实现为 **`answer` 内嵌逻辑**）；负责在 **`self._executor`/`self._mcp_session` 已由同一块内前置步骤赋值**之后，**幂等**拉取 **`list_tools`** 并写入 **`_tools_by_name`**（若已缓存且会话仍有效可早退）。

**单次 `answer` 与连接寿命**：实现可 **(i)** 将上述 **`async with` 嵌套包住 `answer` 全文**（一问一答即一连）；或 **(ii)** 在编排器上实现 **`async def __aenter__` / `__aexit__`**（或 **`aclose`**），使多次 **`answer`** 复用同一会话（自定，须在整仓一致）。

**`config` 来源与形状**：实现从 **`agent_rag/config/settings.yaml`** 读取后，将 **`rag_agent`** 段（或与同文件顶层 **`llm` / `embedding` / `vector_store` 等**合并后的 `dict`）作为 **`RagOrchestrator(config, *, long_term_collection=..., ...)`** 的 **`config`**；YAML 中已列示 **`routing_skill_path`**、**`routing_hint_max_chars`**（§4 **`load_routing_hint`**：`0` 等表示不裁切）、**`mcp.stdio`**、**`mcp_client`**、**`executor`**（含 §3.2 表：**`max_retries` / `backoff_base_sec`** 与 **`fill_arguments_max_llm_retries` / `fill_arguments_retry_backoff_sec`** 等）、**`evaluator` / `generator` / `orchestrator`**（终稿门禁与合成）、可选 **`memory` / `context`**，键名须与加载代码一致。

> **§1 / §2 与 §7**：**具体传参与 API 调用顺序**以 §7 表 **`__init__`**、**`answer`** 为准；本段为总览——§1 存储句柄由 **`RagOrchestrator.__init__`** **自建或经关键字参数注入**；**`answer`** 步骤 1 / 6 **须**显式调用 **`retrieve_context` / `get_relevant_context` / `add_short_term` / `update_context`**。

### §7 与 §5、§6 调用对齐（实现时核对）

- §7 `RagOrchestrator.answer` 中应调用 **`await self._generator.run_subtask(query, task, self._executor, self.get_input_schema, self._evaluator, tool_names=list(self._tools_by_name.keys()))`**（**`tool_names`** 供 **`choose_next_action`** 校验 **`tool_name`**；首参为全局用户问题，与 §6 `global_query` 一致）。**`session_id`** 须贯穿 §1/§2 读写（见 §7 表）。
- `SubtaskResult["observation_for_replan"]` 在 `status=="needs_replan"` 时应非空，并作为**子任务级** `replan(..., observation=...)` 的主要材料之一（可与 MCP 错误摘要合并，实现自定）。
- **全局终稿门禁**失败时，**优先**使用 **`GlobalAnswerReadiness["observation_for_replan"]`**（及 **`suggested_retrieval_changes`**）驱动 **`replan`**；与子任务级观测可拼接，**控长**见 **`global_replan_observation_max_chars`**。
- **`_synthesize_final_answer`** 仅在 **`sufficient_for_answer=True`**（或配置允许不足证据仍作答）后调用；默认路径**不是**简单 `join(draft_text)`。
- **`SubtaskResult["task_id"]`**：**必填**，由 **`run_subtask`** 从 **`task["id"]`** 复制；§7 **`subtask_results`** 与 **`_build_final_evidence_bundle`** 依赖该字段分段，**勿**仅在编排层事后补键（除非 **`run_subtask`** 未实现时的临时兜底）。
