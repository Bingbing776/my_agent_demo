# 代码生成转义问题修复说明

## 问题描述

在 `meta_harness_rag` 生成代码时，LLM 返回的 Python 代码被错误地转义了：
- 换行符 `\n` 被写成了字面的 `\\n`
- 中文字符被转义成 Unicode 编码（如 `\\u521d\\u59cb\\u5316` 而不是 "初始化"）
- 导致生成的 Python 文件无法正常运行

## 根本原因

1. LLM 被要求返回 JSON 格式：`{"final": {"test_source": "Python代码"}}`
2. LLM 正确地将 Python 代码转义成 JSON 字符串（符合 JSON 规范）
3. 但是在某个环节，字符串被**双重转义**了
4. 写入文件时没有进行足够的反转义处理

## 修复方案

### 修改文件 1: `harness/evaluator/runtime.py`

**位置**: 第 937-943 行的 `_write_test_via_fc` 方法

**修改内容**:
- 添加了多次解码逻辑（最多 3 次）
- 检测字符串中是否包含 `\n`、`\t`、`\u` 等转义序列
- 使用 `codecs.decode(text, 'unicode_escape')` 进行解码
- 如果解码后字符串不再变化，停止解码

### 修改文件 2: `harness/generator/runtime.py`

**位置**: 第 155-169 行的 `write_draft_to_file` 方法

**修改内容**:
- 添加了相同的多次解码逻辑
- 在写入文件之前对代码进行反转义处理

## 测试验证

运行测试脚本验证修复：
```bash
cd meta_harness_rag
python test_decode_logic.py
```

预期结果：
- ✅ 双重转义的字符串能正确解码
- ✅ 换行符正确还原
- ✅ 中文字符正确还原
- ✅ 正常字符串不受影响

## 使用说明

修复后，重新运行 `python main.py --max-tasks 3`，生成的代码文件应该：
1. 包含正确的换行符（不是 `\\n`）
2. 包含正确的中文字符（不是 `\\uXXXX`）
3. 可以被 Python 解释器正常解析
4. 可以通过 pytest 测试

## 注意事项

1. 此修复是**防御性的**，即使 LLM 返回正常格式的代码，也不会破坏它
2. 最多尝试解码 3 次，避免无限循环
3. 如果解码失败，保持原样，不会导致程序崩溃
4. 同时修复了 Evaluator 和 Generator 两个模块

## 后续改进建议

1. **根本解决方案**: 找出为什么 LLM 返回的字符串会被双重转义
2. **日志清理**: 确保工具调用日志不会被写入源代码文件
3. **验证机制**: 在写入文件前验证代码的语法正确性
4. **单元测试**: 为解码逻辑添加单元测试
